## What this project does

MoviesDB is an end-to-end data pipeline that takes the publicly available IMDb dataset — millions of rows of raw TSV files — and transforms it into a structured, queryable resource. The pipeline ingests, cleans, models, and loads the data into a relational database, then extracts a curated subset of Marvel Cinematic Universe films and exposes them through an interactive graph interface.

The goal was to build something that mirrors a real-world data engineering workflow: raw data in, clean analytical layer out, with automation, testing, and reproducibility at every step.

## Relational Database Design

The foundation of the project is a normalized MySQL schema that mirrors the IMDb data model. Five core tables capture titles, people, crew relationships, ratings, and episode metadata, plus a reference table of Marvel films used for filtering.

## SQL Analytics Layer

The most technically involved part of my work was writing the SQL extraction logic. The challenge is that IMDb data doesn't label films by franchise — there's no "Marvel" column. Identifying MCU content requires pattern matching across titles, genres, and years, while filtering out false positives.

### Marvel Reference Layer

A new `marvel_movies` table is generated by scraping the official Marvel Cinematic Universe film list from Wikipedia. This table contains a clean, curated list of known Marvel titles.

-  Removes fragile regex-based filtering
-  Enables deterministic joins 
-  Makes queries simpler, faster, and easier to maintain
-  Improves reproducibility (no hidden pattern-matching logic)
-  Allows easy updates when new Marvel movies are released


The extraction produces normalized CSVs — movies, actors, characters, and four relationship tables (plays, appears_in, directs, produces), ready for direct import into Neo4j.

## Automated Pipeline

The workflow is orchestrated by a Python script that:

1. Downloads and decompresses IMDb data dumps from the official source
2. Creates a MySQL Docker container and loads the raw data
3. Runs the SQL extraction queries and exports CSVs

## Neo4j Graph Layer

The graph database models the Marvel universe as nodes (Person, Character, Movie) and relationships (PLAY, APPEAR_IN, DIRECTED, PRODUCED). A Streamlit application provides an interactive front end where users can explore the graph — filtering by actor, movie, or relationship type — with Pyvis rendering the network visualization in the browser.

## What I'd Do Next

### Analyse the data 

So far, this has been an engineering focused project. The next objective is to understand how user ratings may be related to film genres or release periods.

### Automated data refresh

Set up a scheduled workflow (e.g. GitHub Actions) to:

- Pull new IMDb datasets  
- Refresh the `marvel_movies` table  
- Re-run the pipeline every 2 months in sync with new releases  

---

### Cloud-native pipeline (Colab + DuckDB)

Build a fully cloud-based version of the pipeline:

- Use Google Colab as execution environment  
- Process data using DuckDB (no MySQL / Docker needed)  
- Export CSVs directly from DuckDB
